# Docs Guide

Last updated: 2026-03-09

This file is a navigation guide, not a full file index.

The goal is:
- Help a new person understand how the docs are organized.
- Point them to the right starting document for a topic.
- Avoid turning this file into a long list that goes stale.

## How To Use This Folder

Read this file first, then jump to the section that matches your task:
- Product intent or original inputs: go to `reference/`
- Current implementation plans: go to `working/plans/`
- Test / quality / debugging notes: go to `working/quality/`
- Historical specs kept only for traceability: go to `archive/`
- Firebase backend source-of-truth docs: go to `LoveSaving/Firebase/docs`

## Folder Map

### `reference/`
Use this for external source material and original inputs.

Contains:
- proposal PDFs
- wireframes
- other non-working-source artifacts

Start here if you want to answer:
- What was originally requested?
- What did the early product framing look like?

### `working/plans/`
Use this for active implementation thinking.

Contains:
- feature specs
- rollout plans
- implementation plans
- AI-assisted workflow guides

Start here if you want to answer:
- How is a feature supposed to work now?
- What is the current intended implementation?
- What decisions were made during active development?

### `working/quality/`
Use this for validation, testing, and operational notes.

Contains:
- test coverage notes
- logging/debugging notes
- quality decisions

Start here if you want to answer:
- How is this feature tested?
- What known tradeoffs or quality decisions exist?
- How do I debug or inspect this locally?

### `archive/`
Use this only when you need history or old context.

Contains:
- outdated specs
- replaced documents that are kept for traceability

Do not start here unless you are explicitly looking for:
- prior versions
- naming history
- decisions that were later replaced

## What Should Go In This README

This file should contain only:
- folder meanings
- topic entry points
- a small number of canonical starting documents
- rules for where new docs should live

This file should not contain:
- every markdown file in the repo
- temporary working notes
- old one-off investigation docs
- duplicate summaries of documents that already exist

## Where New Docs Should Go

Use these rules:
- External input or source artifact: `reference/`
- Active design / implementation spec: `working/plans/`
- Test strategy / logging / verification notes: `working/quality/`
- Replaced or historical material: `archive/`

If a document becomes the main entry point for a major topic, add a link here.
If it is only a supporting note, do not add it here.

## Naming Convention

- Date suffix: `_YYYY-MM-DD`
- `ARCHIVE` in filename means historical and non-current
- Prefer names that describe topic + role, not author or session
